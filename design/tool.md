# Tool Abstraction Design

## Overview
The `Tool` trait serves as a unified abstraction for capabilities that Large Language Models (LLMs) can invoke. This design ensures that tools are defined once and can be used across different providers (e.g., OpenAI function calling, Anthropic tool use) by converting the internal representation to the provider-specific format.

## Core Trait Definition

The `Tool` trait is designed to be asynchronous and thread-safe.

```rust
use async_trait::async_trait;
use serde_json::Value;
use anyhow::Result;

#[async_trait]
pub trait Tool: Send + Sync {
    /// Returns the unique name of the tool (e.g., "get_weather", "search_web").
    /// This name is what the LLM uses to invoke the tool.
    fn name(&self) -> &str;

    /// Returns a description of what the tool does.
    /// This helps the LLM understand when to use the tool.
    fn description(&self) -> &str;

    /// Returns the JSON Schema for the arguments the tool accepts.
    /// This defines the structure of the input the LLM must provide.
    fn parameters(&self) -> Value;

    /// Executes the tool logic with the given arguments.
    /// Returns an ExecuteResult which can be Success or Failure.
    async fn execute(&self, args: Value) -> Result<ExecuteResult>;
    
    /// Returns the independent definition of the tool.
    fn definition(&self) -> ToolDefinition {
        ToolDefinition {
            name: self.name().to_string(),
            description: self.description().to_string(),
            parameters: self.parameters(),
        }
    }
}
```

## Function Details

### 1. `name`
- **Purpose**: Unique identifier for the tool.
- **Usage**: Used in the API call to the LLM provider to register the tool.

### 2. `description`
- **Purpose**: Explains the tool's functionality to the model.
- **Usage**: Included in the system prompt or tool definition sent to the LLM.

### 3. `parameters`
- **Purpose**: Defines the input schema (JSON Schema).
- **Usage**: Validates the arguments generated by the LLM and informs the LLM of the expected format.

### 4. `execute`
- **Purpose**: The actual implementation of the tool's logic.
- **Input**: `serde_json::Value` (arguments).
- **Output**: `Result<ExecuteResult>` (the enum result of the tool execution).
- **Behavior**: Should capture domain-level errors (like "file not found") as `ExecuteResult::Failure`, allowing the LLM to see the error. System-level errors can be returned as `Err`.

### 5. `definition`
- **Purpose**: Returns a provider-agnostic representation of the tool.
- **Usage**: Used to serialize the tool for API requests or configuration.

## Data Types

### ExecuteResult
Encapsulates the outcome of a tool execution as an enum.

```rust
use serde::{Deserialize, Serialize};
use serde_json::Value;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum ExecuteResult {
    /// The execution was successful. Contains the result data.
    Success(Value),
    /// The execution failed. Contains the error message or details.
    Failure(String),
}
```

### ToolDefinition
This struct represents a tool independently of any specific model provider. It contains all the necessary metadata for an LLM to understand and use the tool.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ToolDefinition {
    pub name: String,
    pub description: String,
    pub parameters: Value, // JSON Schema
}
```

### ToolCall
Represents a request from the LLM to execute a tool.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ToolCall {
    pub id: String,
    pub name: String,
    pub arguments: Value,
}
```

### ToolOutput
Represents the result of a tool execution to be sent back to the LLM.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ToolOutput {
    pub id: String, // Matches the ToolCall id
    pub content: String, // Stringified JSON or text result
}
```

## Extension Points

To add a new tool (e.g., `Calculator`):
1. Create a struct `Calculator`.
2. Implement the `Tool` trait.
3. Define the parameters schema (e.g., using `schemars` or raw JSON).
4. Implement the `execute` method with the calculation logic, returning `ExecuteResult`.
5. Register the tool in a `ToolRegistry` or pass it to the `Provider`.
